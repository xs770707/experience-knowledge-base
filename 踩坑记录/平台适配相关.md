# 平台适配相关踩坑记录

### 不查经验就动手
- **场景**：执行类似任务时，没查之前的成功路径就直接开干
- **坑**：反复试错，浪费资源，效率低
- **教训**：昨天做对的事今天也要做对
- **正确做法**：动手前先搜经验，看之前有没有成功路径

### 复杂链路可靠性低
- **场景**：通过多步中转实现功能（如本地→服务端→第三方→目标）
- **坑**：任何一环出问题整个链路就断，调试困难
- **教训**：链路越长越不稳定，优先用平台原生能力
- **正确做法**：优先使用平台原生渠道，避免多步中转

### 子任务效率低
- **场景**：简单任务也派子任务，子任务也绕路
- **坑**：子任务重复主任务的错误，效率更低
- **教训**：简单任务自己做，给子任务的指令要明确方法
- **正确做法**：3步以内能搞定的事不派子任务

### GitHub国内访问不稳定
- **场景**：国内访问github.com、github.com/new等页面偶发无法打开
- **坑**：页面打不开就卡住，无法判断是网络问题还是操作问题
- **教训**：不要硬刷新死等，换个时间或网络再试
- **正确做法**：先用API接口（如 api.github.com）确认仓库状态，排除操作问题；网页打不开时等一段时间重试，或换网络环境

### GitHub Token类型选错
- **场景**：GitHub Developer Settings里有Fine-grained tokens和Tokens (classic)两种
- **坑**：Fine-grained tokens权限配置复杂，需要逐项设置仓库访问权限和细粒度操作权限，容易漏配导致操作失败
- **教训**：简单上传文件用classic就够了，Fine-grained适合精细权限控制场景
- **正确做法**：生成Personal Access Token时选Tokens (classic)，勾选repo权限即可满足文件上传需求
